Connection pooling
-> maximumPoolSize per server
maximum connection one instance can have at a time with DB
suppose DB supports 100 connection general rule is 0.6*{maxconnection}/numberOfservers

-> minimumIdle
minimum no. of connections per instance created at the time of initialisation
generally 20% of max number of connections

-> connectionTimeOut
time for which a request waits for a connection and gives exception if pool is exhausted
2000ms - for low latency system
5sec - for batch jobs

-> maxLifeTime
Max life of a connection before it is recycled.
generally 30 min

-> idleTimeout
remove the connections from pool that are idle but maintain minimuIdle
Spiky traffic → keep higher (1–5 min)
Stable traffic → keep lower (30–60 sec)

So as soon as service starts minimum connections are established as per requests more connections are added to the pool with limit maximum pool size and on connection timeout request
throw Error applying backpressure, maxLifeTime recycle the connections before DB server timeout and the connections that are idle for a while are removed as per idleTimeout config.

deadlocks: generally databases try to avoid deadlocks by running a deadlock detection algorithm before any transaction could accuire lock [to avoid lock transaction kills itself]

shared lock 
Select * from ... for shared
-> other transactions can have shared lock on same rows
T1 -> row 1,2,3: read and modify but for modify it has to be converted to exclusive lock for that all other threads have to release there shared lock
T2 -> row 3,4,5 : read
read throuput increases.

exclusive
Select * from ... for Update
untill T1 is complete T2 can't even read.

skip lock
it's just a extension
Select * from ... for Update skip locked
it will skip all those rows that are already locked by some other thread but get lock on rest

NOWAIT
Select * from ... for Update NOWAIT
if any row is already locked kill self transaction

So if we have isolation levels in ACID then why this locking is needed as those isolation level are on transactional level which imapact scheduling of transaction like serializable
would not allow other transaction to run[in this case extra locking is not needed] which most of system won't have because of high throughput needs other repeatble read, read commit
read uncommit these don't block tranactions to access same row so they have edgecases which could leed to wrong/stale reads in tranctions which is fine with ACID as we chose that
that isolation level it is Consistent at db level but from application perspective we have got wrong data that's why this locking mechanism is important.

-> Replication in PostgreSQL is primarily log-based. The primary writes changes to WAL, which is streamed to replicas. Replicas apply those logs to stay in sync.
Depending on configuration, replication can be synchronous or asynchronous, trading off between consistency and latency.

steps 
replica establishes a connection with primary and tells I have logs untill this point send afterwards
Your Java app sends a POST request to insert "apple".
The Primary database writes "apple" to its own local disk.
The Primary database immediately replies to your Java app with a "Success" message.
A fraction of a second later, the Primary drops that unread log data onto the streaming connection, sending it to the Replica.
